  Methodology caveat (read before trusting the SUPPORTED/TOFU column above):
  on the P0 test platform (macOS, Chromium-based automated browser), every
  column — including "(e) KFGQPC Arabic Symbols 01", which has no @font-face and no
  such font installed, and even generic "(a) serif" — rendered every single codepoint
  as a real ligature glyph, not tofu. This is because the browser silently substitutes
  glyphs from whatever Arabic-capable system font IS installed whenever the requested
  font-family lacks the glyph (standard CSS font-fallback behavior). That is actually
  representative of what an end user would SEE (a real glyph, not a box) — but it means
  this test cannot isolate "does font X specifically carry this glyph" from "did the OS
  quietly substitute a different font." The canvas-width TOFU heuristic in this page's
  script has the same blind spot for the same reason. Practical implication: on this
  Mac, the Unicode-14 honorific block (U+FD40-FD4F) already renders correctly with
  zero custom fonts shipped, via macOS's own Arabic font fallback. That is NOT proof
  that Windows or older Android will do the same — those platforms have historically
  had weaker/no coverage of this brand-new Unicode 14 block, and verifying them requires
  a human tester on that actual OS (this spike cannot reach a Windows/Android renderer).
